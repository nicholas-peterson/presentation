Defect Zero!

...or, how I learned to stop worrying and embrace quality.

Who Am I?

  • Learned to program with C++
  • Love Relational Databases more than is healthy.

Me in natural habitat.

Where is this talk headed?

I like to know it's a rollercoaster before I get in line.

  • Defects
  • Complexity
  • Quality

What is a Defect?

A shortcoming, imperfection, or lack.

When we say, 'Defect Free Software'. We're using a shorthand for saying that we want our software without shortcomings or imperfections.

Where do software defects come from?

  • Complexity produces software defects

  • If something is simple to learn and understand, it tends to be simple to alter.
  • If something is complex to learn and understand, it tends to be complex to alter.

Complexity

'Controlling complexity is the essence of computer programming.' -Brian Kernighan

Types of Complexity:

  • Essential Complexity: Directly related to the problem, absolutely necessary to solve the problem.
  • Accidental Complexity: Not related to the problem, generally comes from tools.

If you changed your entire architecture, essential complexity would remain unaltered, but accidental complexity may vary significantly.

Some Real World Examples:

Accidental Complexity

  • Null Checks
  • Required Syntax
  • Memory Management

Essential Complexity

  • Balancing Financial Transactions
  • Payment Processing
  • Formula Calculating Monthly Bill

Quality

Software with no defects that is still 'fit to purpose', is high quality software.

Since reducing complexity lowers the defect rate, doing so while remaining 'fit to purpose' must raise quality.

Quality = Fit to Purpose / Defect Rate

Tests

Cease dependence on inspection to achieve quality. Eliminate the need for massive inspection by building quality into the product in the first place. -William E. Deming

  • Testing finds defects in our software.
  • The defects probably exist because of complexity.
  • Testing is a way of measuring quality, not creating it.
  • Focus on decreasing complexity rather than increasing tests to deal with complexity
  • Tests are a way to mitigate areas of high complexity that are essential.

Complexity vs Ease

  • Often we think of tools (languages, libraries, frameworks, programs), as reducing complexity.

That's not exactly true, for instance. SQL Server is extremely complex, and brings with it many things that we'll never have need for. But using it is easier than building one ourselves, and likely better.

We've added massive complexity in return for ease.

Complexity vs Ease cont.

This applies to almost all tools. Often tooling we use is combating the symptoms of complexity without treating the root cause, complexity itself.

For example:

FXCop is a great tool for enforcing a consistent style of coding, but isn't it's very use a sign that the tool in question (in this case let's say it's C#) is allowing people to use it in undesirable ways?

Simplify Essential Complexity.

There are many ways to attempt this, but it falls mostly outside of the scope of this talk

I encourage anyone here not intimately familiar with Domain Driven Design to look into it. Notice the word complexity in the tagline of the book? Not a coincidence...

Reducing Accidental Complexity

  • Reduce lines of code
  • Reduce number of tools used to solve a problem

Great paper on this subject to look into it 'Out of the Tar Pit' by Ben Moseley and Peter Marks. They trace software complexity through great minds like Brooks and Dijkstra.

Terse, High-Level Languages Can Lower Accidental Complexity

  • LISP derivatives (Clojure, Racket, Scheme)
  • ML derivatives (Standard ML, Ocaml, F#)
  • Erlang ( Erlang, Elixir)
  • Other notables (Scala, Rust)

Often able to describe a solution in less lines of code, with less ceremony.

What about C#?

Low Complexity Possible Through C#?

In some ways, C# is much less complex than certain languages

C# is easily less complex than two extremely common languages:

  • C++ (has macros, ability to inline C, manual memory management...)
  • Javascript

But there are still 'features' of C# that makes it less easy to quickly reason about. They add incidental complexity, let's look at some examples...

Is the following true or false?

var cust1 = new Customer(99, "J Smith");
var cust2 = new Customer(99, "J Smith");
cust1.Equals(cust2);

Who Knows?

  • If Customer has implemented IEquatable it could be true.
  • The point is you have to delve into Customer to understand what this code will do.
  • Is there any chance cust1 and cust2 are actually different in reality?
  • Ideally comparisons by value would be the default.

What does this print?

var cust = new Customer();
Console.WriteLine(cust.Address.Country);

Depends if the Address property is null or not. Hopefully the constructor initialized Address and the Country property.

The point is that objects can exist in an invalid state.

What does this output?

var repo = new CustomerRepository();
var customer = repo.GetById(42);
Console.WriteLine(customer.Id);

Well, the value of customer could be:

  • A regular, initialized Customer (fingers crossed!).
  • A default Customer object.
  • Customer could be null.
  • The ID could be null.
  • An Exception could be thrown.

Wouldn't it be nice if we didn't have to worry about null?

Will this return an int?

try {
Int32.Parse("ABCxxx");
}
catch (Exception ex) {
   //exception code
}

No, obviously it will throw an exception.

What is an improvement? How about:

int result = 0;
bool parsed = false;
parse = Int32.TryParse("ABCxxx", out result);

Now my code always returns true/false (no exceptions), and I can check the bool before using the output integer.

I've made the possible cases explicit, and the code easier to understand because it isn't full of exception handling code.

Imperative Statements

Imagine if SQL didn't exist and every 'query' resulted in statements like:

foreach (Customer c in Customers)
{
    decimal sum = 0.00;
    foreach (Order o in Orders)
    {
        if (o.CustomerId == c.CustomerId)
        {
            sum += o.Total
        }
    }
}

Statements VS Expressions

Almost all operations in C# use statements. Some things like LINQ are expressions, what's the difference?

  • Statements manipulate state -> cause side effects to produce a result.
  • Expressions can be composed of other expressions -> produce a value.

Statements usually assign their result to a variable, which can change over time.

Lets look at some C# with statements, then replace some of the statements with expressions.

For Loop:

int i;
int length;
var numArray = new int[] { 1, 2, 3 };
int sum;

length = numArray.Length;
for (i = 0; i < length; i++)
{
    sum += numArray[i];
}
Note the placeholder variables, what is their initial value?
explicit control of length/iteration, could run forever.

Foreach Loop:

var numArray = new int[] { 1, 2, 3 };
int sum = 0;

foreach (var i in numArray)
{
    sum += i;
}

LINQ Expression

var numArray = new int[] { 1, 2, 3 };
var sum = array.Sum();

More on Expressions

  • Expressions are composable, you can place an expression into another expression.
  • Statements use side effects to produce results, but C# doesn't guarantee the return type.
  • The move from statements to expressions removes possibilities for errors.
  • Wouldn't it be nice if the entire language used expressions?

Immutable Data Types

C# encourages the use of mutability almost everywhere.

That means it's really easy to create changes, but difficult to track down the source of a change. As soon as you start contending with concurrency or parallelism, it can become a nightmare.

Wouldn't it be nice if your language pushed you down the right path from day one?

High Order Functions

Treating functions as a value is useful.

Allows use to pass functions to other functions. Or return a function from another function.

C# actually does allow this, but it isn't very idiomatic.

Also, most libraries don't do it, so you fight against the tide.

Currying and Partial Application

Allows you to partially specify parameters to a function. It produces a new function with those parameters statically set.

imagine a function like the following

1: 

let add x y = x + y

This function is called 'add' and takes 2 parameters, x and y. It then sums them and returns the new value. We could also add this definition.

1: 

let addFive = add 5

We partially filled in a call to 'add' with the first parameter set to 5.

Now we can call addFive without creating a new duplicate definition.

Algebraic Types

In C#, types are either built into the langauge as keywords, or defined as classes.

Unfortunately, they have pretty high ceremony, and combining them results in a lot more code.

Algebraic types allows one to add or multiple 'types' (not values) together to produce new types.

consider these types:

1: 
2: 
3: 

type EmailAddress = string
type CustomerName = string
type ContactInfo = EmailAddress * CustomerName

contactInfo is the equivalent of a C# tuple that has two properties, one of type emailAddress and one of type customerName.

Algebraic Types Continued..

But we could also 'sum' types together to make types that can be either one or the other.

1: 
2: 
3: 

type ContactInfo =
|Complete of emailAddress * customerName
|Incomplete of emailAddress

Now we have a type that could be either Complete or Incomplete, and we can make code respond accordingly.

Pattern Matching

So we have a ContactInfo type, how does using that look in practice?

Match Expression:

1: 
2: 
3: 

match contact with
| Complete email,name -> sendEmail email name  
| Incomplete email -> sendEmail email "Sir or Madame"

We tested for the type of the contact and then reacted accordingly.
